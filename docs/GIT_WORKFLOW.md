# 🌳 Git 워크플로우 가이드 (Fork 기준)

> 깃이 처음이어도 **이 문서 순서대로만** 따라오면 됩니다.
> 막히면 조수연에게 물어보세요!

---

## 0. 🗺 전체 그림 먼저

우리 저장소에는 브랜치가 3종류 있습니다.

```
main          배포용        🔒 잠김 (PR + 승인 3명)
 └ dev        통합용        🔒 잠김 (PR + 승인 3명)
    └ dev_이름  내 작업공간   ✅ 자유롭게 push
```

**핵심 규칙 3가지**

1. 코드는 **항상 내 브랜치(`dev_이름`)에서만** 작성한다
2. `dev` 에 넣을 때는 **직접 push 불가 → 반드시 PR(Pull Request)** 로
3. 작업 전에는 **항상 `dev` 최신 내용을 내 브랜치로 먼저 가져온다**

> 💡 3번이 제일 중요해요. 안 하면 나중에 충돌로 고생합니다.

## 1. 📅 하루 작업 흐름 (한 사이클)

### ① 작업 시작 전 — dev 최신화

| 순서 | Fork에서 |
|---|---|
| 1 | 상단 **`Fetch`** 클릭 (서버 최신 정보 가져오기) |
| 2 | 왼쪽 **`dev`** 더블클릭 (체크아웃) |
| 3 | 상단 **`Pull`** 클릭 |
| 4 | **내 브랜치**(`dev_이름`) 더블클릭 (체크아웃) |
| 5 | `dev` 우클릭 → **`Merge dev into dev_이름`** |

> ✅ 이제 내 브랜치가 dev 최신 상태를 품고 있습니다. 여기서부터 작업 시작!

### ② 코드 작업
평소처럼 IDE에서 개발합니다.

### ③ 커밋

| 순서 | Fork에서 |
|---|---|
| 1 | 왼쪽 상단 **`Local Changes`** 클릭 |
| 2 | 변경 파일 선택 → **`Stage`** (또는 `Stage All`) |
| 3 | 아래 칸에 커밋 메시지 작성 |
| 4 | **`Commit`** 클릭 |

**커밋 메시지 양식**
```
Summary      ❇️[feat] 여행 목표 등록 API
Description  - travel VO/Mapper 작성
             - 등록 서비스 구현
```
> 타입: `❇️[feat]` `🐞[fix]` `💄[style]` `🔨[refactor]` `🔧[chore]` `📝[docs]` `🔍[test]` `🚧[setting]`

### ④ Push
상단 **`Push`** 클릭 → **내 브랜치**로 올라가는지 확인 후 Push

### ⑤ PR 올리기 → [2번 항목](#2--pr-pull-request-올리기) 으로

## 2. 🔀 PR (Pull Request) 올리기

### 언제 올리나?

> **기능 하나 완성될 때마다!** (우리 팀 규칙)

| ✅ 좋은 PR | ❌ 나쁜 PR |
|---|---|
| "여행 목표 등록 API" 하나 | "여행 기능 전부" |
| 파일 3~10개, 300줄 내외 | 파일 80개, 3000줄 |
| 하루~이틀 작업 | 2주치 몰아서 |

> 크면 리뷰어가 제대로 못 봐서 **리뷰가 의미 없어지고**, 충돌도 심해집니다.

### 어떻게 올리나?

**Fork에서 시작**
1. 왼쪽 **내 브랜치 우클릭** → **`Create Pull Request on 'origin'`**
2. → 브라우저가 열리며 GitHub PR 작성 화면으로 이동

**GitHub에서 마무리**

3. **방향 확인** ⭐ 가장 중요
   ```
   base: dev   ←   compare: dev_이름
   ```
   > ⚠️ `base`가 **`main`이 아니라 `dev`** 인지 꼭 확인! (기본값이 main이라 바꿔야 함)

4. **제목**: 커밋과 같은 양식 → `❇️[feat] 여행 목표 등록 API`
5. **본문**: 템플릿이 자동으로 뜹니다. 빈칸만 채우세요.
6. 오른쪽 **`Reviewers`** 에서 팀원 지정
7. **`Create pull request`**

### 그다음은?
- **승인 3명** 을 받아야 병합 버튼이 열립니다 (자기 PR은 자기가 승인 불가)
- 승인 완료 → **`Merge pull request`** → **`Create a merge commit`** 선택 → `Confirm merge`

> ⚠️ `Squash and merge` 는 쓰지 마세요. 커밋 ID가 바뀌어서 개인 브랜치를 계속 쓰는 우리 방식과 충돌이 납니다.

### 병합된 다음 (중요!)
다음 작업 전에 **①번(dev 최신화)을 다시** 해주세요. 안 하면 다음 PR에서 충돌 납니다.

## 3. 👀 코드 리뷰 하는 법

PR이 올라오면 알림이 옵니다. 다음 순서로 리뷰해주세요.

1. PR 링크 열기
2. 상단 **`Files changed`** 탭 클릭 (변경된 코드가 보임)
3. 코멘트를 남기고 싶으면 → 해당 **줄 왼쪽 `+` 버튼** 클릭 → 내용 작성
4. 오른쪽 위 초록색 **`Review changes`** 클릭
5. 3가지 중 선택 → **`Submit review`**

### 리뷰 3종류

| 종류 | 뜻 | 병합 영향 | 언제 |
|---|---|---|---|
| 💬 **Comment** | 의견·질문 | 없음 (중립) | 궁금한 점, 가벼운 제안 |
| ✅ **Approve** | 승인 | **승인 1개 추가** | 문제 없을 때 |
| 🔴 **Request changes** | 수정 요청 | **병합 차단** 🚫 | 꼭 고쳐야 할 때만 |

> ⚠️ **`Request changes` 는 신중하게!**
> 하나만 있어도 병합이 막히고, **요청한 본인이 다시 Approve로 바꿔줘야** 풀립니다.
> 오타·사소한 제안은 `Comment` + `Approve` 로 해주세요.

### 리뷰할 때 볼 것
- 컨벤션을 지켰는가 (`CONVENTIONS.md`)
- 남의 파트 코드를 건드리지 않았는가
- `console.log`, 안 쓰는 주석이 남아있지 않은가
- 이름이 명확한가 (`data1`, `temp` ❌)

## 3-1. 🔴 내 PR이 `Request changes` 를 받았다면

기분 나빠하지 마세요! 코드를 같이 좋게 만드는 과정입니다 😊

### 해결 순서

| 순서 | 할 일 |
|---|---|
| 1 | PR 의 **`Files changed`** 탭에서 지적받은 내용 확인 |
| 2 | **내 브랜치에서 코드 수정** |
| 3 | Fork 에서 **Stage → Commit → Push** |
| 4 | 각 코멘트에 **답글 + `Resolve conversation`** 클릭 |
| 5 | `Reviewers` 옆 **🔄 버튼**으로 재리뷰 요청 |
| 6 | 리뷰어가 `Approve` 로 바꿔주면 해제 ✅ |

### ⭐ 꼭 기억할 것

**PR 을 새로 만들 필요가 없습니다!**
같은 브랜치에 push 하면 **기존 PR 에 자동으로 반영**됩니다. (제일 많이 헷갈리는 부분)

커밋 메시지 예시
```
🐞[fix] 리뷰 반영 - 목표 조회 시 null 체크 추가
```

### ⚠️ 우리 저장소 설정 때문에 주의할 점

| 설정 | 의미 |
|---|---|
| **새 커밋 시 승인 초기화 = ON** | 코드를 수정해 push 하면 **기존 승인 3개가 모두 사라집니다** → 다시 3명에게 받아야 함 |
| **코멘트 해결 필수 = ON** | 코멘트를 `Resolve` 하지 않으면 **병합 버튼이 열리지 않습니다** |

> 💡 그래서 **PR 은 완성도를 갖춰서 올리는 게 좋아요.** 올린 뒤 계속 고치면 승인을 계속 다시 받아야 합니다.

### 이럴 땐?
- **지적이 이해 안 됨** → 코멘트로 물어보세요. *"이 부분 왜 그렇게 하는 게 좋을까요?"*
- **리뷰어가 안 풀어줌** → 슬랙에서 멘션하거나 조수연에게 문의

## 4. 💥 충돌(Conflict) 났을 때

충돌은 **잘못한 게 아니라 자연스러운 일**입니다. 당황하지 마세요!

### 충돌이 나는 이유
내가 작업하는 동안 **다른 사람이 같은 파일을 고쳐서 dev에 먼저 병합**했을 때 생깁니다.

### 해결 순서

1. 내 브랜치에서 `dev` 우클릭 → **`Merge dev into dev_이름`**
2. 충돌 파일이 표시됨 → 오른쪽 **`Resolve`** 클릭
3. 머지 창에서 **왼쪽 = dev(남의 코드)** / **오른쪽 = 내 코드**
4. **둘 다 살펴보고** 최종 코드를 만든다
5. **`Commit`** → **`Push`**

### ⚠️ 충돌 해결 원칙
- **무조건 한쪽을 선택하지 마세요.** 남의 코드를 지우면 그 사람 기능이 사라집니다 😱
- 보통은 **양쪽 다 살려야** 합니다 (내 함수 + 남의 함수 둘 다 남기기)
- 애매하면 **해당 코드 작성자에게 물어보세요!** (슬랙에 "이 부분 충돌났는데 어떻게 합칠까요?")

## 5. 🧳 작업 중인데 dev를 받아야 할 때 (Stash)

작업 중이면 브랜치 이동이 안 됩니다. 이럴 때 **Stash(임시 보관)** 를 씁니다.

1. 상단 **`Stash`** 클릭 → 설명 입력 (예: `login form working`)
   > 작업 내용이 안전하게 보관되고, 작업 공간이 깨끗해집니다
2. `dev` 체크아웃 → **`Pull`**
3. 내 브랜치 체크아웃 → `dev` 우클릭 → **`Merge dev into dev_이름`**
4. 왼쪽 **`Stashes`** 에서 아까 저장한 것 우클릭 → **`Apply`**
5. 충돌 나면 [4번](#4--충돌conflict-났을-때) 대로 해결
6. **`Commit`** → **`Push`**

## 6. 📋 치트시트

```
[작업 시작]
Fetch → dev 체크아웃 → Pull → 내 브랜치 체크아웃 → dev를 내 브랜치에 Merge

[작업 후]
Stage → Commit → Push (내 브랜치)

[dev 에 반영]
Fork: 내 브랜치 우클릭 → Create Pull Request
GitHub: base=dev 확인 → 제목·본문 → Reviewers 지정 → Create
        → 승인 3명 → Merge pull request (Create a merge commit)

[병합 후]
다시 [작업 시작] 부터 (dev 최신화!)

[작업 중 dev 받아야 하면]
Stash → dev Pull → 내 브랜치에 dev Merge → Stash Apply → 충돌 해결 → Commit → Push
```

## 7. ❌ 자주 하는 실수

| 실수 | 결과 | 예방 |
|---|---|---|
| `dev` / `main` 에 직접 push | **차단됨** (에러) | 항상 내 브랜치 → PR |
| PR 만들 때 `base`를 `main`으로 | 배포 브랜치에 바로 감 | **base = dev** 확인! |
| dev 최신화 없이 계속 작업 | 나중에 대형 충돌 | 작업 전 항상 Fetch+Pull |
| 2주치 몰아서 PR | 리뷰 불가, 충돌 지옥 | 기능 하나마다 PR |
| 충돌 때 한쪽 통째로 선택 | 남의 코드 삭제 😱 | 양쪽 다 살리기 |
| 남의 브랜치에 push | 혼란 | 내 브랜치만! |

## 8. 🆘 이럴 땐?

**Q. push 했는데 `rejected` 에러가 나요**
→ `dev`나 `main`에 push하려 한 겁니다. 내 브랜치로 push하세요.

**Q. push 했는데 권한 없다고 나와요 (403)**
→ GitHub 저장소 초대를 수락했는지 확인하세요.

**Q. 잘못 커밋했어요**
→ 아직 push 전이면 Fork에서 커밋 우클릭 → `Reset`. push 후면 조수연에게 문의!

**Q. 브랜치를 잘못 만들었어요**
→ 그냥 두고 내 브랜치(`dev_이름`)로 체크아웃해서 작업하세요.

**Q. 뭘 해야 할지 모르겠어요**
→ [6번 치트시트](#6--치트시트) 순서대로만 하시면 됩니다!
